Reseller API reference

{{icon "arrow-right"}}back to settings

The complete JSON REST surface resellers use to manage their own users, orders, DIDs, KYC bundles, ledger, CDRs, and live calls. Every endpoint is bearer-token authenticated and scoped server-side to the calling key's reseller. Generate keys per reseller from /resellers.

QUICKSTART

Every endpoint accepts and returns JSON. The base URL is your public DIDStorage host. Replace YOUR_KEY with the secret from the /resellers page.

curl https://didstorage.example/api/v1/me \
  -H "Authorization: Bearer YOUR_KEY"

AUTHENTICATION

Every request must carry a bearer token in the Authorization header. Keys are bcrypt-hashed server-side and looked up on every request; revoking a key in the admin GUI takes effect immediately. Missing or invalid keys return 401.

Authorization: Bearer YOUR_KEY

ERROR SHAPE

Errors return a JSON object with a single error field plus the appropriate HTTP status.

{
  "error": "user not in your reseller"
}
CodeMeaning
400Bad request, validation failed, malformed JSON
401Missing or invalid bearer token
402Insufficient balance to charge NRC
403Resource exists but does not belong to your reseller
404Resource not found within your reseller scope
409Conflict (e.g. DID already assigned, order not in editable state)
500Server error

DATA ISOLATION

Every query is scoped to the reseller_id derived from your API key. You can never read, write, or even discover another reseller's data. The platform also guarantees these resources are NEVER exposed via the reseller API:

Never exposed to the reseller API:
  • Supplier names, supplier IPs, supplier hostnames (sanitised in SIP traces to did:E164)
  • Other resellers' users, orders, ledger, CDRs, KYC bundles
  • Admin email addresses or admin-side compliance actions
  • Reserved-DID metadata (admin-only routing, including audio reservations)
  • Asterisk channel names, internal admin-action stamps on calls
  • Raw supplier-side IP addresses on CDRs (the To-URI's host is rewritten to the DID in trace exports)
  • Other resellers' API keys, brand names, hostnames

HOW CARRIERS REACH US

Resellers don't talk to the SIP signalling path directly. The flow is one-way inbound:

supplier (your carrier) ──INVITE──> Asterisk (UDP/TCP 5060)
                                      │
                                      ├─ AGI: /opt/didstorage/scripts/dids-authorize.py
                                      │       ↓
                                      │   POST /sipctl/authorize  (internal, shared-token auth)
                                      │       ↓
                                      │   {decision, route_kind, route_target, max_seconds, ...}
                                      │
                                      ├─ Dial(PJSIP/outbound/sip:user@host, max_seconds)
                                      │
                                      └─ hangup-handler  ──> POST /sipctl/cdr  (internal)
                                                              ↓
                                                          ledger debit + CDR row written

The reseller API surfaces the AFTER-effects of this flow: CDRs, ledger entries, live-call state, supplier-side denial outcomes. It does NOT expose /sipctl/* (which is reserved for Asterisk itself, behind a shared X-DIDS-Auth token), nor any of the underlying supplier identity (IPs, hostnames, supplier names).

curl

curl https://didstorage.example/api/v1/me -H "Authorization: Bearer YOUR_KEY"

USERS

Customer-level entities owned by your reseller. Balance, channel cap, KYC, and SIP peers all hang off a user.

GET/api/v1/users

List every user under your reseller, newest first. Capped at 500 rows per call.

Response

{
  "users": [
    {
      "id": 12,
      "external_id": "ACME-2026-00123",
      "label": "Acme HQ",
      "contact_email": "ops@acme.example",
      "status": "active",
      "balance_cents": 12450,
      "global_channel_cap": -1,
      "created_at": "2026-01-04T12:00:00Z",
      "active_orders": 4
    }
  ]
}
POST/api/v1/users

Create a new user under your reseller. external_id is your reference and is required; it is what you'll use to find the user later via /api/v1/users/by-external/{ext}.

Body

{
  "external_id": "ACME-2026-00123",
  "label": "Acme HQ",
  "contact_email": "ops@acme.example",
  "global_channel_cap": -1
}

Response

{ "id": 12, "external_id": "ACME-2026-00123" }
GET/api/v1/users/{id}

Fetch one user by id.

GET/api/v1/users/by-external/{ext}

Fetch one user by your external_id. Useful when you only have your own reference.

POST/api/v1/users/{id}/topup

Credit the user's balance and append a topup ledger entry. Returns the new balance. Amounts are in integer cents (USD).

Body

{ "amount_cents": 5000 }

Response

{ "balance_cents": 17450 }

LEDGER

Every credit and debit on the user's balance, oldest entry first when sorted. Source of truth for billing reconciliation. The cached balance_cents on the user record is the running sum.

GET/api/v1/users/{id}/ledger

SIP PEERS

Registered SIP accounts under one of your users. The username is what you reference as the route_target on an order with route_kind = "sip_account". The digest secret (ha1) is NEVER returned; you set it server-side via POST and the platform discards the plaintext immediately. POST and DELETE also rewrite /etc/asterisk/pjsip_users.conf and run pjsip reload so the new peer is registerable immediately.

GET/api/v1/users/{id}/sip-accounts

List SIP peers for one user.

Response

{
  "sip_accounts": [
    { "id": 3, "user_id": 12, "username": "acme01", "realm": "didstorage.local",
      "created_at": "2026-04-10T12:00:00Z" }
  ]
}
POST/api/v1/users/{id}/sip-accounts

Create a SIP peer. Username 1-32 chars [a-zA-Z0-9_-]; reserved Asterisk section names and -auth / -aor suffixes are rejected. Password ≥ 8 chars. Asterisk reload happens in the background.

Body

{ "username": "acme01", "password": "a-strong-secret" }

Response

{ "id": 3, "username": "acme01", "realm": "didstorage.local" }
DELETE/api/v1/users/{id}/sip-accounts/{said}

Remove a SIP peer. Asterisk reload happens in the background. Calls already in progress on the peer continue until natural hangup.

ORDERS

A per-DID rental owned by one user. Holds the route (where calls land), channel cap, anniversary day for monthly billing, and optional KYC bundle. NRC is charged on create; MRC + per-channel fees on each anniversary.

GET/api/v1/orders

List every order under your reseller, newest first.

POST/api/v1/orders

Rent a DID for one of your users. The DID must currently be available in /api/v1/dids. NRC is charged from the user's balance atomically; if balance is insufficient you get 402 and nothing is written. If you supply a kyc_bundle_id that's already approved the order starts active, otherwise kyc_pending.

Body

{
  "user_id": 12,
  "did_id": 401,
  "channel_count": 2,
  "route_kind": "sip_uri",
  "route_target": "sip:helpdesk@acme.example:5060",
  "anniversary_day": 5,
  "kyc_bundle_id": null
}

Response

{
  "order_id": 92,
  "status": "active",
  "nrc_charged_cents": 500
}
GET/api/v1/orders/{id}

Fetch one order by id.

Response

{
  "id": 92, "user_id": 12, "did_id": 401, "did_e164": "442038968001",
  "status": "active", "channel_count": 2,
  "route_kind": "sip_uri", "route_target": "sip:helpdesk@acme.example:5060",
  "next_billing_at": "2026-06-05T00:00:00Z",
  "kyc_bundle_id": 7, "kyc_status": "approved"
}
PATCH/api/v1/orders/{id}

Update the route, channel count, or attached KYC bundle on an existing order. Only valid while the order is active, kyc_pending, or quarantined. All fields are optional; only provided fields change. Attaching an approved bundle to a kyc_pending order automatically promotes it to active.

Body

{
  "route_kind": "sip_uri",
  "route_target": "sip:newhelpdesk@acme.example",
  "channel_count": 4,
  "kyc_bundle_id": 7
}

Response

{ "status": "updated", "route_kind": "sip_uri",
  "route_target": "sip:newhelpdesk@acme.example", "channel_count": 4 }
POST/api/v1/orders/{id}/cancel

Cancel an order. The DID returns to the available pool immediately. Cancellation is irreversible; create a new order to re-assign the DID.

GET/api/v1/orders/{id}/cdrs

GET/api/v1/cdrs/{call_id}/sip-trace

Return the SIP wire-trace observed on the platform for one of your CDRs. The response is sanitised: every supplier IP is rewritten to did:E164, and the platform's public IP is rewritten to your brand_name. You never see who the upstream carrier was. Accepts either the sanitised prefix or the raw prefix@host:port form.

KYC BUNDLES

KYC documents are uploaded by you on behalf of a user and approved (or rejected) by the platform admins. A bundle is required before an order leaves kyc_pending.

POST/api/v1/users/{id}/kyc-bundles

Create a new bundle for one of your users. Bundle starts pending; admins approve in the admin GUI.

Body

{ "type": "person", "info": { "legal_name": "...", "address": "..." } }
GET/api/v1/users/{id}/kyc-bundles

List bundles for a user.

GET/api/v1/kyc-bundles/{bid}

Get a bundle's details and its uploaded documents (filename + size only, never the file bytes).

POST/api/v1/kyc-bundles/{bid}/documents

Upload a KYC document. Multipart form. Field file is the binary, field kind classifies it (id_front, id_back, passport, address_proof, company_registration, other). Max 25 MB.

curl

curl -H "Authorization: Bearer YOUR_KEY" \
     -F "kind=id_front" -F "file=@id.jpg" \
     https://didstorage.example/api/v1/kyc-bundles/7/documents
GET/api/v1/kyc-bundles/{bid}/documents/{did}/download

Download a stored document.

CSV EXPORTS

CSV mirrors of the JSON list endpoints for accounting / compliance. All accept optional from + to date-range query params (UTC, YYYY-MM-DD).

GET/api/v1/users/{id}/export/cdrs.csv

All CDRs for one user.

GET/api/v1/users/{id}/export/ledger.csv

All ledger entries for one user.

GET/api/v1/orders/{id}/export/cdrs.csv

All CDRs for one order.
